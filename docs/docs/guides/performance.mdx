# Performance & Memory

This guide covers performance optimization and memory management in Mystral Native.js.

## V8 Heap Memory

When using V8 builds, complex scenes may encounter out-of-memory (OOM) errors due to V8's default heap size limits.

### Symptoms

- Application crashes after running for a few minutes
- Error messages about heap allocation failures
- Performance degradation over time

### Solution: Increase V8 Heap Size

You can increase V8's heap limit using the `V8_FLAGS` environment variable:

```bash
# macOS/Linux: Set max heap to 4GB (in bytes)
V8_FLAGS="--max-old-space-size=4096" mystral run game.js

# Or export it for the session
export V8_FLAGS="--max-old-space-size=4096"
mystral run game.js
```

Common heap sizes:
- `2048` (2GB) - Good for most games
- `4096` (4GB) - Complex scenes with many textures
- `8192` (8GB) - Very large worlds (Sponza, etc.)

### Windows

```powershell
$env:V8_FLAGS="--max-old-space-size=4096"
.\mystral.exe run game.js
```

## Shadow Memory Usage

Shadows can use significant GPU memory. For simple scenes or low-memory situations, disable shadows:

```javascript
import { Engine } from '@mystral/core';

const engine = new Engine({
  shadowsEnabled: false  // Saves ~160MB VRAM
});
```

Or with custom shadow settings:

```javascript
const engine = new Engine({
  shadow: {
    enabled: true,
    resolution: 512,  // Lower = less memory (default: 1024)
    maxLights: 1      // Fewer shadow-casting lights
  }
});
```

## Hot Reload for Development

Use watch mode to avoid restart overhead during development:

```bash
mystral run game.js --watch
```

This watches for file changes and automatically reloads the script, keeping WebGPU context initialized.

## Performance Tips

### 1. Minimize Allocations

Avoid creating objects in your render loop:

```javascript
// Bad - creates new vector every frame
function update() {
  const velocity = new Vector3(1, 0, 0);
  player.position.add(velocity);
}

// Good - reuse objects
const velocity = new Vector3(1, 0, 0);
function update() {
  player.position.add(velocity);
}
```

### 2. Use Instancing

For many similar objects, use instanced rendering:

```javascript
// Instead of 1000 separate meshes
const mesh = new InstancedMesh(geometry, material, 1000);
```

### 3. Level of Detail (LOD)

Reduce detail for distant objects:

```javascript
const lod = new LOD();
lod.addLevel(highDetailMesh, 0);    // Close
lod.addLevel(mediumDetailMesh, 50); // Medium distance
lod.addLevel(lowDetailMesh, 200);   // Far
```

### 4. Texture Optimization

- Use compressed textures (KTX2, Basis) when possible
- Match texture resolution to object screen size
- Use mipmaps for distant objects

## Profiling

### Reproducible Runtime Benchmark

The CLI profiler measures a fixed number of frames after an explicit warmup and can export the result as JSON:

```bash
mystral run benchmarks/runtime-core.js --no-sdl \
  --benchmark 240 --warmup 120 \
  --benchmark-output runtime-core.json
```

The report includes frame-phase percentiles, frame-budget misses, V8 heap snapshots, Worker activity, and native job-system metrics. Keep the script, warmup, measured frame count, build type, and machine unchanged when comparing commits.

### Live MCP Frame Sampling

When a game is running with `--debug-port`, the MCP tool
`mystral_performance_sample` profiles an exact number of live frames. It returns
the frame profile, runtime/WebGPU snapshots before and after the window, recursive
numeric deltas, and the optional `performance.live` semantic snapshot.

```bash
mystral run game.js --headless --debug-port 9222
bun tools/mystral-mcp-server.ts --port 9222
```

This is the preferred path for an agent comparing a specific scene or camera
state: inspect the semantic scene, request a fixed frame sample, make one explicit
change, then sample the same number of frames again. Cumulative WebGPU totals are
included for context; use the returned deltas for the measured window.

### Native Job-System Streaming Benchmark

Local asset reads made through `fetch()` run on the native priority job system and complete on the main thread. Use the included workload to measure that path:

```bash
mystral run benchmarks/job-system-streaming.mjs --no-sdl \
  --benchmark 240 --warmup 120 \
  --benchmark-output job-system-streaming.json
```

The exported `jobs` object contains:

- `submitted`, `completed`, `cancelled`, `failed`, and `rejected` counters for the measured interval
- `queueEnd`/`queuePeak` and `inFlightEnd`/`inFlightPeak` for detecting saturation
- `queueWait` and `execution` latency distributions with mean, p50, p95, p99, and maximum values

A rising `queueWait.p95Ms` with stable execution time indicates contention. A non-zero `rejected` count means the in-flight backpressure limit was reached. For gameplay, keep frame-critical work short and split large streaming/decode operations into cancellable jobs.

### WebGPU Bridge Profiling

Native builds expose cumulative per-operation bridge measurements through:

```javascript
const stats = globalThis.__mystralWebGpuStats?.();
console.log(stats?.profiledBridge);
```

Each operation reports `calls`, `totalNanoseconds`, and transferred `bytes`. Take snapshots before and after a fixed frame window and subtract them; reading cumulative totals directly mixes startup uploads with steady gameplay. The profiled operations cover command encoder creation/finalization, render and compute passes, queue submits, buffer/texture writes, draw commands, bindings, and render-bundle execution.

Timing is measured inside the native binding, including argument conversion and the backend call. It does not include JavaScript work performed before entering the binding, so compare it with the application's total CPU submission time.

### Batched WebGPU Buffer Writes

Mystral exposes the non-standard `GPUQueue.writeBufferBatch()` extension for renderers that issue many small `writeBuffer()` calls. The argument is a flat array with five entries per write:

```javascript
const writes = [
  cameraBuffer, 0, cameraData, undefined, undefined,
  lightBuffer, 0, lightData, 0, lightData.length,
];

if (typeof device.queue.writeBufferBatch === 'function') {
  device.queue.writeBufferBatch(writes);
} else {
  device.queue.writeBuffer(cameraBuffer, 0, cameraData);
  device.queue.writeBuffer(lightBuffer, 0, lightData, 0, lightData.length);
}
```

Each group is `[buffer, bufferOffset, data, dataOffset, size]`; `dataOffset` and `size` follow the standard `writeBuffer()` element-count rules and may be `undefined`. Writes execute in array order. Flush a pending batch before `submit()`, `writeTexture()`, `copyExternalImageToTexture()`, or `onSubmittedWorkDone()` so queue ordering remains unchanged.

The bridge profiler reports these calls separately as `queueWriteBufferBatch`, including their total bytes. This extension reduces JS/C++ crossings; it does not combine the underlying WebGPU queue writes.

### Non-blocking GPU and Image Work

On Dawn builds, the standard asynchronous WebGPU methods are connected to the
native callbacks and settle their Promises on the runtime thread:

```javascript
const pipeline = await device.createRenderPipelineAsync(descriptor);

device.queue.submit([commandEncoder.finish()]);
await device.queue.onSubmittedWorkDone();

await readbackBuffer.mapAsync(GPUMapMode.READ);
const result = readbackBuffer.getMappedRange();
```

`createComputePipelineAsync()` behaves the same way. These methods do not spin
or sleep on the JavaScript thread, so timers, workers, input, and frames can keep
progressing while Dawn compiles a pipeline or waits for GPU work. Keep the
synchronous `createRenderPipeline()` and `createComputePipeline()` methods for
small startup-only pipelines or code that requires an immediate result.

`createImageBitmap()` also decodes PNG, JPEG, and WebP data on the shared native
job system. It is safe to start several texture decodes and await them together:

```javascript
const bitmaps = await Promise.all(
  textureUrls.map(async (url) => createImageBitmap(await (await fetch(url)).blob()))
);
```

The Promise continuations still run on the main JavaScript isolate. Only native
decode/compile/wait work runs concurrently; JavaScript game-state mutation must
still use Workers or `WorkerPool` when it needs parallel CPU execution.

### Frame Time

Log frame timing to identify bottlenecks:

```javascript
let lastTime = performance.now();
let frameCount = 0;

function render() {
  const now = performance.now();
  const delta = now - lastTime;
  lastTime = now;

  frameCount++;
  if (frameCount % 60 === 0) {
    console.log(`Frame time: ${delta.toFixed(2)}ms (${(1000/delta).toFixed(1)} FPS)`);
  }

  // Your render code...

  requestAnimationFrame(render);
}
```

### Memory

Check V8 heap usage periodically:

```javascript
// Note: This requires --expose-gc flag
if (typeof gc === 'function') {
  gc(); // Force garbage collection
}
console.log('Heap used:', process.memoryUsage?.().heapUsed / 1024 / 1024, 'MB');
```

## Known Limitations

1. **V8 GC Pauses** - Large heaps may cause occasional frame drops during garbage collection
2. **Texture Memory** - GPU memory usage is separate from V8 heap; monitor with platform tools
3. **macOS Audio Shutdown** - Process may exit with code 137 due to SDL3/CoreAudio interaction (audio works correctly during runtime)
