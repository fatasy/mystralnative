# Workers and Shared Memory: LLM Handoff

This document is the implementation guide and architectural contract for MystralNative's parallel JavaScript Workers and shared-memory APIs. It is written so that another LLM or contributor can continue the work without assuming browser behavior that the runtime does not implement.

## Scope and mental model

MystralNative runs the game entry point in one JavaScript engine instance (the **main isolate**). Every `Worker` runs on a real OS thread with its own JavaScript engine instance and its own heap. JavaScript objects, closures, module state, and garbage-collected heaps are never shared between those instances.

Cross-thread communication has three paths:

1. `postMessage` clones a versioned structured subset, including binary views,
   through bounded native queues.
2. A `postMessage` transfer list moves plain `ArrayBuffer` ownership and detaches
   the sender.
3. `SharedBuffer`, `SharedTable`, and `SharedQueue` expose explicitly shared, typed memory without copying the backing bytes.

```text
main JS isolate
  | Worker constructor / postMessage
  v
runtime-owned WorkerRegistry
  | owns WorkerThread objects and native message queues
  +--------------------+--------------------+
  v                    v                    v
OS thread 1          OS thread 2          OS thread N
JS isolate 1         JS isolate 2         JS isolate N
  \_____________________|____________________/
                 SharedBufferStorage
          (native bytes with shared lifetime)
```

This is intentionally not a shared JavaScript heap. The runtime shares storage; each isolate creates its own `SharedArrayBuffer` and typed-array wrapper over the same native bytes.

## Worker lifecycle

The current lifecycle is:

1. Runtime startup creates one `WorkerRegistry` and one `SharedBufferRegistry`. They belong to that runtime instance; neither is a process-global singleton.
2. `setupURL()` installs the public `Worker` class and the private native hooks `__mystralWorkerCreate`, `__mystralWorkerPostMessage`, and `__mystralWorkerTerminate` in the main isolate. A first-level Worker installs the same registry hooks with a Worker-owned child registry.
3. `new Worker(source, options)` asks `WorkerRegistry` to create a `WorkerThread`.
4. `WorkerThread` starts a `std::thread`, creates a new engine of the configured kind, installs shared-memory APIs and Worker globals, then evaluates either a classic script or an ES module.
5. Incoming messages and monotonic timer deadlines wait on a `condition_variable`. There is no polling loop.
6. The Worker parses a message, reconstructs transferred buffers, calls its `message` listeners and `onmessage`, then queues outbound messages produced by `postMessage`.
7. The owning isolate drains native outbound queues and invokes `__mystralDispatchWorkerMessage`. The main runtime does this once per event-loop turn; a Worker parent is awakened through its condition variable when a child publishes output, so nested Workers do not require polling.
8. `terminate()`, runtime shutdown, and hot reload request stop, interrupt currently executing JavaScript on V8, recursively stop and join descendants, and only then destroy the owning JavaScript engine.

Worker callbacks and native JobSystem handlers acquire the same runtime CPU
budget only while executing. Waiting isolates and queued native jobs consume no
permit.

Internally the lifecycle is explicit: `Created`, `Starting`, `Running`, `Stopping`, `Stopped`, or `Failed`. A Worker also emits an internal exit message so the main-side facade removes Workers that call `close()` or fail during startup.

In `--no-sdl` mode, a live Worker counts as pending event-loop work. The runtime stays alive until the Worker calls `close()`, exits after a failure, or is terminated. Call `worker.terminate()` or `await pool.close()` when a long-lived WorkerPool is no longer needed.

Each Worker has these globals:

- `self`
- `onmessage` and `onerror`
- `addEventListener` and `removeEventListener`
- `postMessage`
- `close`
- `importScripts`
- `setTimeout`, `clearTimeout`, `setInterval`, and `clearInterval`
- `queueMicrotask`
- `runNativeTask` (also exported by `mystral/native-tasks`)

An internal ready event only marks the owner-side `Worker` facade as ready. It is not emitted to the user's `onmessage` handler.

### Script and module sources

A Blob or Blob URL is treated as a classic script. A path/URL is treated as a module and is loaded by MystralNative's module resolver. Module imports resolve relative to their referrer. Source paths begin at the runtime root/current working directory.

Do not assume full browser URL or Worker semantics. `postMessage` implements a
versioned structured subset for JSON-compatible values, `ArrayBuffer`, TypedArray,
`DataView`, and shared handles. It is not the complete browser structured-clone
algorithm.

## Message contract

Ordinary control values keep JSON semantics. Binary values are encoded as small
typed metadata records while their backing buffers travel as native attachments;
TypedArray elements are not expanded into JSON properties.

Each direction is bounded by default to 1,024 queued user messages, 16 MiB per
message, and 64 MiB accumulated in the queue. Input `postMessage()` returns a
deterministic native status that the public facade maps to `RangeError` for a
full queue or oversized message. Worker-side output uses the same limits and
throws inside the Worker when saturated. Messages are never silently added
beyond the configured bounds; internal ready/exit lifecycle signals are reserved
so saturation cannot hide shutdown.

Limits can be selected per Worker. Values are positive safe integers and one
message cannot be configured larger than the accumulated queue budget:

```js
const worker = new Worker('./simulation.worker.mjs', {
    type: 'module',
    maxMessages: 64,
    maxMessageBytes: 256 * 1024 * 1024,
    maxQueuedBytes: 384 * 1024 * 1024,
});
```

Supported examples include numbers, strings, booleans, `null`, arrays, plain
objects, shared handles, `ArrayBuffer`, every TypedArray constructor available in
the engine, and `DataView`. Functions, cyclic graphs, symbols, prototypes, and
`undefined` as a complete payload are not preserved. Transfer-list entries must
still be plain `ArrayBuffer` objects, not views.

```js
const bytes = new ArrayBuffer(1024);
const ids = new Uint32Array(bytes, 128, 16);
const header = new DataView(bytes, 0, 32);
worker.postMessage({ ids, header }, [bytes]);
// bytes.byteLength === 0
```

The receiver reconstructs the view constructor, byte offset, length, and backing
aliasing. If the backing is in the transfer list, V8 detaches and moves it without
copying. If it is absent, the sender clones that backing once and keeps the
original usable; multiple views of the same backing share one clone. Queue byte
accounting includes both metadata and attached bytes. Duplicate entries,
non-`ArrayBuffer` entries, and buffers already transferred through the public API
fail explicitly.

Raw `SharedArrayBuffer` values and views are rejected. Use a `SharedBuffer`,
`SharedTable`, or `SharedQueue` handle when the same storage must stay accessible
from multiple isolates.

Shared-memory handles are JSON objects whose underlying buffer marker is:

```js
{ $mystralSharedBuffer: numericId }
```

The JSON reviver recognizes that marker and attaches a local `SharedArrayBuffer` wrapper to the same native storage. Only the small descriptor crosses the queue; the backing bytes are not copied.

## Shared-memory ownership and lifetime

`SharedBufferRegistry` allocates zero-initialized `SharedBufferStorage` and assigns a numeric ID. The registry keeps strong ownership so another isolate can attach after receiving a handle. Engine-specific backing-store callbacks also retain shared ownership while a JavaScript buffer wrapper exists.

`SharedBuffer.release()` removes the registry's ownership for that ID. Existing attached wrappers can keep the storage alive, but no new isolate should be expected to attach after release. Application code must therefore release only after every required attachment has completed and the shared region is being retired.

Engine status:

| Engine | Worker isolation | Forced termination | Shared memory |
| --- | --- | --- | --- |
| V8 | Supported | `TerminateExecution()` | Supported with an external `v8::SharedArrayBuffer` backing store |

## Public shared API

Import the virtual module from every module that defines or attaches a schema:

```js
import { SharedBuffer, SharedTable, SharedQueue, SharedCommandBuffer } from 'mystral/shared';
```

The module is virtual. There is no package file for an application to copy, and a Worker knows a schema only because its own module imports the schema definition. A schema name alone does not generate a JavaScript binding.

### SharedBuffer

```js
const bytes = SharedBuffer.allocate(4096);
const view = new Uint32Array(bytes.buffer);

worker.postMessage({ buffer: bytes });

// Equivalent explicit descriptor:
const handle = bytes.handle();
const attached = SharedBuffer.attach(handle);
```

Public members are `id`, `byteLength`, `buffer`, `handle()`, `toJSON()`, and `release()`.

### SharedTable

`SharedTable` is a fixed-capacity structure-of-arrays store. Define the same schema in an ordinary shared application module and import that definition from both main and Worker modules:

```js
// character-rows.mjs
import { SharedTable } from 'mystral/shared';

export const CharacterRows = SharedTable.define('game.characters.v1', {
  templateIndex: 'u16',
  settlementId: 'u32',
  householdRef: 'u32',
  birthDay: 'i32',
  energy: 'f32',
});
```

```js
// main.mjs
import { CharacterRows } from './character-rows.mjs';

const characters = CharacterRows.create({ capacity: 200_000 });
characters.length = 100_000;
characters.templateIndex[0] = 12;

worker.postMessage({ characters: characters.handle() });
```

```js
// character-worker.mjs
import { CharacterRows } from './character-rows.mjs';

self.onmessage = ({ data }) => {
  const characters = CharacterRows.attach(data.characters);
  for (let i = data.begin; i < data.end; i++) {
    characters.energy[i] += 1;
  }
  postMessage({ type: 'done' });
};
```

This answers the important ownership question: `CharacterRows` is application code, defined once in `character-rows.mjs` and imported independently into each isolate. The native runtime provides `SharedTable`; it does not know the game's character schema.

Supported field types are `u8`, `i8`, `u16`, `i16`, `u32`, `i32`, `f32`, and `f64`. Fields appear directly as typed arrays on the attached instance.

The first 64 bytes are a control header. Its relevant `Int32` slots are:

| Index | Meaning |
| --- | --- |
| 0 | 32-bit schema hash |
| 1 | capacity |
| 2 | current length |

The schema ID, ordered field names/types, capacity, backing-buffer size, and header are validated during attachment. Changing a schema requires a new versioned `schemaId`, for example `game.characters.v2`.

`SharedTable` does not resize transparently. `CharacterRows.resize(characters, { capacity })` allocates a new table, copies `min(oldLength, capacity)` rows field by field, and returns the migrated instance. The old table remains valid. Use a game-controlled safe point: stop writers, call `resize`, distribute the new handle, wait for acknowledgements, then call `oldTable.release()`. Shrinking truncates `length` to the new capacity.

### SharedQueue

`SharedQueue` is a bounded **single-producer/single-consumer** (SPSC) ring for typed records:

```js
const Commands = SharedQueue.define('game.commands.v1', {
  kind: 'u16',
  entity: 'u32',
  value: 'f32',
});

const commands = Commands.create({ capacity: 4096 });
commands.push({ kind: 1, entity: 42, value: 3.5 }); // false when full

const command = commands.pop(); // null when empty
```

It uses `capacity + 1` physical slots so full and empty states are distinguishable. Header slots are schema hash, logical capacity, producer head, and consumer tail. The producer writes a record before publishing the new head atomically; the consumer reads only published records and then advances the tail.

Do not use one queue with multiple producers or multiple consumers. Create separate SPSC queues or implement a separately specified MPSC/MPMC primitive with the required algorithm and tests. Do not silently broaden this class's contract.

### SharedCommandBuffer

`SharedCommandBuffer` is the pool-oriented alternative to an MPMC queue. It
allocates one single-writer lane per Worker and merges the lanes only after a
barrier:

```js
const Commands = SharedCommandBuffer.define('game.simulation-commands.v1', {
  entity: 'u32',
  kind: 'u16',
  value: 'f32',
});

const commands = Commands.create({ workerCount: 4, laneCapacity: 4096 });
commands.push(workerIndex, { entity: 42, kind: 1, value: 3.5 });

// Owner only, after every Worker completed:
const committed = commands.drain({ sortBy: ['entity', 'kind'] });
```

`push()` returns `false` instead of overwriting a full lane. `drain()` visits
lanes in Worker-index order and resets them. With dynamic scheduling, use a
stable `sortBy` key for globally deterministic commit order; equal keys retain
their prior lane order, so use a unique tie-breaker when that matters.

## Deterministic WorkerPool

The virtual `mystral/worker-pool` module exports `WorkerPool`,
`exposeWorkerTasks`, and `transferResult`.

- `await WorkerPool.create(url, options)` creates a fixed set of module Workers and waits for every task module to become ready. Direct construction is unsupported.
- Startup has a 30-second default bound (`startupTimeoutMs`) so a module that never exposes handlers cannot stall initialization forever.
- `pool.parallelFor(name, data, { length, grainSize })` lets Workers dynamically claim small chunks through a shared atomic counter.
- `pool.parallelFor(name, data, { length, schedule: 'static' })` assigns stable contiguous ranges; lower Worker indexes receive remainder rows.
- `pool.forEach(name, data, options)` is the dynamic mutation-only API. It skips chunk results and only the last Worker publishes one completion message through a shared atomic barrier.
- Every execution method returns a Promise, task handlers may be `async`, and `close()` provides an awaitable shutdown API.
- One round executes at a time; additional rounds wait in a bounded FIFO instead of failing merely because the pool is busy.
- Dynamic tasks can request an ordered owner-side reducer (`sum`, `count`, `min`, `max`, or a local function).
- Shared-handle dynamic rounds use one native serialization/broadcast. `parallelFor()` returns chunk results; `forEach()` returns one completion message for the whole barrier.
- Duplicate/stale Worker results do not advance the barrier.
- A task error rejects the round and terminates the pool.
- `pool.resizeTable(definition, table, options)` delegates to explicit table migration and rejects while the pool is busy. It cannot guard unrelated Workers, so the global safe-point rule still applies.

`WorkerPool` is available in the main isolate and in a first-level Worker. The
default maximum depth is two, counting the main isolate as depth zero. This
supports a simulation-owned pool while preventing task Workers from recursively
creating more Workers:

```text
main/render isolate -- commands/projections --> simulation Worker (Game owner)
                                               | safe point + deterministic round
                                               v
                                      WorkerPool child isolates
                                               |
                                      SharedTable / optional SPSC queues
```

Every parent isolate owns the registry for its direct children. All registries in
the tree share the runtime's worker budget and shared-buffer registry. Child
output wakes the parent event loop, errors are delivered to the direct parent,
and parent shutdown recursively interrupts and joins descendants. The runtime
defaults to at most 64 active Workers across the complete tree. Embedders can
change `RuntimeConfig::maxWorkerDepth` and `RuntimeConfig::maxWorkerThreads`.
The runtime also shares one CPU execution budget between Worker callbacks and
native `JobSystem` work. `RuntimeConfig::maxCpuThreads == 0` reserves one logical
CPU for main/render; an explicit positive value overrides it. Pool defaults use
this budget rather than independently consuming every logical CPU.
The nested `Worker` facade accepts module URL strings; inline `Blob` and classic
script sources remain main-isolate-only. `WorkerPool` always uses module URLs.

Worker modules register named task handlers:

```js
import { exposeWorkerTasks, transferResult } from 'mystral/worker-pool';

exposeWorkerTasks({
  update({ data, begin, end, round, workerIndex, workerCount }) {
    // Mutate only the assigned shared-memory partition.
    return { begin, end };
    // For binary output: return transferResult({ bytes }, [bytes]);
  },
});
```

Several systems can share the same persistent isolates:

```js
import { exposeWorkerTasks } from 'mystral/worker-pool';

exposeWorkerTasks({
  economy: updateEconomy,
  demography: updateDemography,
  characterAI: evaluateCharacters,
});
```

Dynamic scheduling changes execution order, not the required result. Give each
chunk disjoint output, derive randomness from stable inputs such as
`(tick, systemId, entityId)`, and commit structural commands in a stable owner
order. Per-chunk transferable buffers are rejected; shared tables and command
lanes are the intended bulk-output path.

All round APIs accept `signal: AbortSignal`. A queued abort rejects immediately.
For an active round, the owner publishes cancellation through the shared control
buffer, Workers stop claiming new chunks, and the Promise rejects only after the
safe barrier. The handler context includes `isCancelled()` so expensive chunks
can return early without abandoning shared-state ownership. This is cooperative
cancellation; `await pool.close()` force-interrupts and joins the pool when a
barrier cannot be awaited.

TypeScript projects can load `types/mystral.d.ts` directly; the root package also
advertises it through its `types` field.

## Native tasks inside a Worker

The virtual `mystral/native-tasks` module exports `runNativeTask(name, data)`.
The public C++ registry is declared in
`include/mystral/workers/native_task.h`:

```cpp
workers::getNativeTaskRegistry().registerTask(
    "game.path-batch",
    [](std::string_view wirePayload, const async::JobContext& context) {
        // Parse handles, run C++ or call a Rust FFI kernel, then return wire data.
        if (context.isCancelled()) {
            return workers::NativeTaskResult::failure("cancelled");
        }
        return workers::NativeTaskResult::success(outputWirePayload);
    });
```

Submission uses the runtime priority JobSystem. A completion mailbox belongs to
the calling Worker, wakes that Worker's condition variable, and resolves the
Promise only on its own V8 thread. Worker shutdown closes the mailbox, cancels
queued work from the same generation, and safely ignores late native completion.
The bridge deliberately accepts no ordinary `ArrayBuffer` attachments; use
value data or shared handles so native lifetime remains explicit.

## Integrating an existing CharacterStore

Keep the store as the domain API and invariant owner. Replace its private typed-array storage with views supplied by an attached `SharedTable`; do not replace all store methods with raw array access throughout the game.

For a store with fields such as `templateIndices: Uint16Array`, `settlementIds: Uint32Array`, `householdRefs: Uint32Array`, and `birthDays: Int32Array`, the direct schema mapping is `u16`, `u32`, `u32`, and `i32`. Other hot simulation fields can be added when they have a clear owner.

Recommended division of responsibility for a main-owned simulation:

- Main thread owns entity creation, deletion, packed-reference allocation, render state, and WebGPU resources.
- Workers read stable identity/liveness metadata and update explicitly assigned row ranges or components.
- Each mutable region has one writer during a simulation phase.
- Workers report completion or compact commands through `SharedQueue`/`postMessage`; the main thread applies structural changes at a safe point.

If the game owns authoritative state in a simulation Worker, keep entity
creation, deletion, packed-reference allocation, and structural changes in that
Worker. The render main isolate still owns WebGPU resources and consumes only
projections. The simulation Worker may use a pool for stable data-parallel phases;
task Workers return results or compact commands, and the simulation owner applies
structural changes after the pool barrier.

If entity references encode index, generation, or namespace bits, continue validating them through the allocator/store contract. Sharing the underlying arrays must not bypass stale-reference checks.

## Concurrency invariants

These rules are part of the implementation contract:

1. Never access a JavaScript object created by one isolate from another isolate.
2. Give each mutable field or row partition one writer per phase.
3. Use Atomics for synchronization words, counters, ownership flags, and publication. Do not make every numeric component atomic by default.
4. Never resize, release, or replace a region while Workers may still attach to or use it.
5. Keep rendering and WebGPU ownership on the main thread until a separate, verified graphics-thread contract exists.
6. Treat `SharedQueue` as SPSC.
7. Give each `SharedCommandBuffer` lane exactly one Worker writer and drain only after the barrier.
8. Treat messages as the documented structured subset, not complete browser
   structured clone; transfer lists still contain only plain `ArrayBuffer` values.
9. Version schemas when their binary layout changes.
10. Terminate/join Workers before destroying their engine or shared registries.

Shared memory removes serialization and copying cost for bulk state, but introduces data races, false sharing, cache-coherency traffic, fixed-capacity planning, and more difficult debugging. Parallel execution is only faster when each task contains enough work to amortize scheduling and synchronization.

## Profiling and fixed benchmarks

The runtime profiler records Workers created during the sample, active Worker
count at sample start/end, processed messages, processed timer
callbacks, rejected messages in four counters (`inputQueueFull`,
`inputMessagesTooLarge`, `outputQueueFull`, and `outputMessagesTooLarge`), largest
observed message, cumulative Worker busy time, input/output queue bytes at sample
end, lifetime queue high-water bytes, and shared-memory bytes at start/end. The
end-of-sample queue values show remaining backlog; the lifetime high-water values
show the worst retained queue even when it drained before sampling ended. Use those
measurements together with frame/tick time; Worker utilization by itself does not
prove a faster game loop. Any rejected message means a producer reached
backpressure and must raise an intentional limit, reduce traffic, batch work, or
use a shared queue. CPU-budget limit/current/peak values are included in runtime
stats and benchmark profiler output so oversubscription control is observable.

The fixed shared-memory benchmark uses 200,000 rows, four Workers, ten warmup samples, and fifty measured samples. It initializes baseline and parallel tables identically, assigns exclusive row partitions, and validates checksum equality before reporting median, p95, and p99.

```bash
mystral run benchmarks/worker-shared.mjs
```

The nested benchmark uses the same workload but places the pool inside an
authoritative simulation Worker and includes the main-to-simulation round trip:

```bash
mystral run benchmarks/worker-nested.mjs --no-sdl
```

The scheduler-v2 benchmark deliberately clusters expensive rows in the last
static partition, then compares fixed partitioning with dynamically claimed
512-row chunks using the mutation-only atomic barrier, and validates equal
checksums:

```bash
mystral run benchmarks/worker-scheduler-v2.mjs --no-sdl
```

The transferable-buffer benchmark moves the same buffer from the main isolate to a Worker and back. It covers 64 KiB, 1 MiB, and 8 MiB payloads, checks sender detachment and sentinel bytes on every round, and reports round-trip median/p95/p99, effective bidirectional throughput, and heap snapshots as JSON:

```bash
mystral run benchmarks/worker-transfer.mjs --no-sdl
```

The machine-readable report is the line prefixed with `[WorkerTransferBenchmark]`. Redirect or filter that line to retain the JSON result in automation. Heap snapshots are observational and do not force garbage collection. Do not change the fixed inputs of either benchmark when comparing runtime changes. Create an additional benchmark if a different workload is needed.

The configurable-limit benchmark verifies 16 MiB, 32 MiB, and 64 MiB typed-view
round trips under an explicit 80 MiB per-message / 96 MiB queued budget:

```bash
mystral run benchmarks/worker-message-limits.mjs --no-sdl
```

## Safe extension guide

When extending this subsystem:

- New table field type: add it to `TYPE_INFO`, define alignment and typed-array behavior, then cover create/attach and schema mismatch.
- New message value category: decide whether it is copied, transferred, or shared; specify lifetime and failure behavior before changing serialization.
- Queue with additional producers/consumers: create a new primitive with an explicit memory-ordering algorithm instead of modifying SPSC semantics in place.
- New resize policy: preserve the explicit safe-point migration protocol; do not mutate a live backing-store size or replace attached views.
- Worker graphics access: keep it out of this subsystem unless device/thread affinity and shutdown ordering have a separate design and validation plan.

## Change checklist

Before considering a Worker/shared-memory change complete, verify:

- Main and Worker isolates still own independent JS heaps.
- Worker shutdown interrupts executing JavaScript on V8 and joins all native threads.
- Input/output queues remain bounded and rejection metrics are observable.
- Shared backing storage cannot be freed while an engine wrapper uses it.
- Schema/layout mismatch fails explicitly.
- Every shared mutable region has a documented writer.
- Queue producer/consumer cardinality matches its contract.
- Correctness is checked with deterministic output/checksum, not timing alone.
- The V8 path is tested when changed.
- Timer callbacks sleep until their deadline and remain interruptible during shutdown.
- WorkerPool partitions and barrier result order remain deterministic.
- Profiler metrics and both fixed benchmarks still work.
- Public docs and this handoff are updated when the contract changes.
