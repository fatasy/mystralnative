# CLI Commands

Complete reference for the Mystral Native.js command-line interface.

## Overview

```
mystral <command> [options]

Commands:
  run <script>      Run a JavaScript/TypeScript file
  compile <entry>   Bundle into a single executable
  --version         Show version
  --help            Show help
```

## mystral run

Run a JavaScript or TypeScript file with WebGPU support.

```bash
mystral run <script.js> [options]
```

### Options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `--width` | number | 1280 | Window width in pixels |
| `--height` | number | 720 | Window height in pixels |
| `--title` | string | "Mystral" | Window title |
| `--headless` | flag | - | Run without displaying a window |
| `--no-sdl` | flag | - | Run without SDL (headless GPU, no window system) |
| `--watch`, `-w` | flag | - | Watch mode: auto-reload on file changes |
| `--screenshot` | string | - | Take screenshot and exit |
| `--frames` | number | 60 | Frames to render before screenshot |
| `--quiet`, `-q` | flag | - | Suppress all output except errors |
| `--debug` | flag | - | Enable verbose WebGPU/shader debug logging |
| `--debug-port` | number | - | Enable debug server on specified port (for testing) |
| `--benchmark` | number | - | Profile exactly this many frames, then exit |
| `--warmup` | number | 120 | Frames to execute before profiling begins |
| `--benchmark-output` | string | - | Write the benchmark report as JSON |

### Video Recording Options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `--video`, `--record` | string | - | Record video to file (WebP or MP4) |
| `--start-frame` | number | 0 | First frame to capture |
| `--end-frame` | number | - | Last frame to capture (required for video) |
| `--video-fps` | number | 60 | Video framerate |
| `--video-quality` | number | 80 | WebP quality 0-100 (higher = better) |
| `--mp4` | flag | - | Convert to MP4 (requires `ffmpeg` on PATH, auto-detected if path ends in .mp4) |

### Examples

```bash
# Basic usage
mystral run game.js

# Custom window size
mystral run game.js --width 1920 --height 1080

# Custom title
mystral run game.js --title "My Game"

# Headless rendering
mystral run game.js --headless

# Take screenshot for CI testing
mystral run game.js --headless --screenshot output.png --frames 120

# Record 5 seconds of video (300 frames at 60fps)
mystral run game.js --headless --video demo.webp --end-frame 300

# Record video with MP4 output (requires FFmpeg)
mystral run game.js --headless --video demo.mp4 --end-frame 600

# Enable verbose debug logging
mystral run game.js --debug
```

### Fixed Runtime Benchmark

Use the repository's stable CPU workload to compare runtime builds. Warmup frames
allow the selected JavaScript engine to optimize the workload before measurements
begin:

```bash
mystral run benchmarks/runtime-core.js \
  --no-sdl \
  --benchmark 600 \
  --warmup 120 \
  --benchmark-output runtime-benchmark.json
```

The profiler reports mean, p50, p95, p99, and maximum time for the complete frame
and for these runtime phases: events, asynchronous work, callbacks, animation frame,
and cleanup. It also records frame-budget misses and JavaScript heap snapshots before
and after the measured window.

Native Worker activity is reported separately as Workers created, messages and
timer callbacks processed, messages rejected, cumulative Worker busy time, input/output queue bytes at the
end of the sample, lifetime input/output queue high-water bytes, and shared-memory
bytes. Rejected messages indicate backpressure rather than silent dropping.
Worker busy time is not added to main-thread frame time because Workers execute
concurrently.

The benchmark workload is versioned and intentionally avoids GPU rendering and
per-frame allocations. Keep its contents unchanged when comparing commits. Use a
separate script with the same CLI options to profile a real game scene.

### Agent and Test Automation

Start a live game with the local WebSocket debug protocol enabled:

```bash
mystral run game.js --headless --debug-port 9222
```

The repository includes a Bun client so an agent or test can control the running
process without a browser:

```bash
bun tools/mystral-debug-client.ts hello --port 9222
bun tools/mystral-debug-client.ts evaluate "globalThis.score" --port 9222
bun tools/mystral-debug-client.ts pause --port 9222
bun tools/mystral-debug-client.ts step 3 --port 9222
bun tools/mystral-debug-client.ts screenshot output.png --port 9222
bun tools/mystral-debug-client.ts logs --port 9222
bun tools/mystral-debug-client.ts agent-list --port 9222
bun tools/mystral-debug-client.ts agent-inspect player --port 9222
bun tools/mystral-debug-client.ts agent-act player.move '{"x":4,"y":2}' --port 9222
bun tools/mystral-debug-client.ts metrics --port 9222
bun tools/mystral-debug-client.ts profile 120 30000 --port 9222
bun tools/mystral-debug-client.ts quit --port 9222
```

Games can expose stable semantic IDs, live snapshots, and safe actions without
coupling the runtime to a particular scene library:

```js
mystralAgent.exposeInspector("player", () => ({
  health: player.health,
  position: { x: player.x, y: player.y }
}), {
  label: "Player",
  kind: "actor",
  description: "Current player state"
});

mystralAgent.exposeAction("player.move", ({ x, y }) => {
  player.moveTo(x, y);
  return { accepted: true };
}, {
  entityId: "player",
  description: "Move the player in world coordinates",
  inputSchema: {
    type: "object",
    required: ["x", "y"],
    properties: { x: { type: "number" }, y: { type: "number" } }
  }
});
```

`exposeInspector()` and `exposeAction()` return cleanup functions. Registering
the same ID again replaces its previous registration, which keeps IDs stable
when game systems are rebuilt. Inspectors and actions are synchronous in this
protocol version; an action can still start asynchronous game work and return
an immediate acknowledgement.

| Protocol method | Purpose |
|-----------------|---------|
| `hello` / `getCapabilities` | Negotiate the protocol version and supported methods |
| `screenshot` | Capture the current WebGPU frame as PNG data |
| `evaluate` | Evaluate JavaScript and return a JSON-safe result, error, and stack |
| `getLogs` | Read buffered startup and runtime console entries |
| `agent.list` | Discover exposed entities, actions, descriptions, and input schemas |
| `agent.inspect` | Read a live JSON-safe snapshot by stable entity ID |
| `agent.act` | Run an explicitly exposed game action with JSON input |
| `metrics.snapshot` | Read current runtime and cumulative WebGPU counters |
| `profile.start` / `profile.stop` | Measure an exact live frame window |
| `pause` / `resume` | Pause or resume game-frame execution while debug I/O remains available |
| `stepFrames` | Execute a bounded number of frames and remain paused |
| `waitForFrame` | Wait asynchronously for an absolute or relative frame with a timeout |
| `keyboard.*`, `mouse.*`, `gamepad.*` | Inject native input through SDL |
| `quit` | Stop the controlled runtime cleanly |

The server only listens on `127.0.0.1`. Protocol messages are JSON objects sent
over WebSocket and use numeric request IDs, allowing multiple clients to reuse
the same request IDs safely.

### MCP Server

The same protocol is available directly to MCP clients through a local `stdio`
server. Start the game first, then configure the client to run:

```bash
bun tools/mystral-mcp-server.ts --port 9222
```

Or use `bun run mcp -- --port 9222`. The safe default tool set exposes status,
semantic discovery, bounded scene inspection, camera state and explicitly enabled
camera actions, game actions, live frame profiling, screenshots, and logs.

Two capabilities require explicit process flags:

```bash
# Adds pause/resume/step/key press/quit
bun tools/mystral-mcp-server.ts --port 9222 --allow-control

# Also exposes arbitrary JavaScript evaluation inside the game
bun tools/mystral-mcp-server.ts --port 9222 --unsafe-evaluate
```

`MYSTRAL_DEBUG_PORT` can provide the port when `--port` is omitted. The MCP server
writes only protocol messages to stdout; diagnostics go to stderr.

## mystral compile

Bundle a game into a single distributable executable.

```bash
mystral compile <entry.js> [options]
```

### Options

| Option | Type | Default | Description |
|--------|------|---------|-------------|
| `--output`, `--out`, `-o` | string | `./<entry-stem>` | Output file path |
| `--include`, `--assets` | string | - | Asset directory to bundle (repeatable) |
| `--root` | string | cwd | Root directory for bundle paths |
| `--bundle-only` | flag | - | Create standalone `.bundle` file instead of executable |

### Examples

```bash
# Compile into a standalone binary
mystral compile game.js --include assets --out dist/my-game
./dist/my-game

# Include multiple asset directories
mystral compile game.js --include assets --include sounds --out dist/my-game

# Create a .bundle file for macOS .app packaging
mystral compile game.js --include assets --bundle-only --out dist/game.bundle
```

### Bundle Mode

The `--bundle-only` flag creates a standalone `.bundle` file containing your JS entry point and assets, without copying the runtime executable. This is used for macOS `.app` packaging where the clean Mach-O binary needs to be code-signed separately:

```bash
# 1. Create the bundle
mystral compile game.js --include assets --bundle-only --out build/game.bundle

# 2. Package as macOS .app (using package-app.sh)
./scripts/package-app.sh \
    --binary build/mystral \
    --name "My Game" \
    --bundle build/game.bundle \
    --output dist
```

At runtime, the binary auto-detects `game.bundle` next to the executable, in `../Resources/` (macOS `.app`), or via the `MYSTRAL_BUNDLE` environment variable.

## Global Options

These options work with any command:

| Option | Description |
|--------|-------------|
| `--version`, `-v` | Print version and exit |
| `--help`, `-h` | Print help message |

## Exit Codes

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Script error or runtime error |
| 137 | Audio shutdown (macOS known issue) |

## Environment Variables

| Variable | Description |
|----------|-------------|
| `MYSTRAL_DEBUG` | Enable debug logging |
| `MYSTRAL_ASSET_PATH` | Override default asset search path |
| `MYSTRAL_BUNDLE` | Path to an external `.bundle` file to load |
| `MYSTRAL_HEADLESS` | Set to `1` to force headless mode |
| `V8_FLAGS` | V8 engine flags (e.g., `--max-old-space-size=4096`) |

## File Types

Mystral Native.js supports:

- `.js` - JavaScript (ES2024)
- `.ts` - TypeScript (transpiled via SWC)
- `.mjs` - ES modules
- `.mts` - TypeScript ES modules
