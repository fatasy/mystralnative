## Native Game Loop

Use `mystral/game-loop` when simulation cadence must be independent from
rendering cadence. The runtime owns the monotonic clock, fixed-step accumulator,
pause, time scale, manual stepping, and bounded catch-up. Rendering continues to
use `requestAnimationFrame`.

```javascript
import { gameLoop } from 'mystral/game-loop';

gameLoop.configure({
  simulationHz: 20,
  maxCatchUpTicks: 4,
  maxFrameDeltaMs: 250,
});

gameLoop.setTickHandler(({ tick, deltaSeconds }) => {
  updateWorld(tick, deltaSeconds);
});

gameLoop.start();

function render() {
  const { interpolationAlpha } = gameLoop.getState();
  renderInterpolatedWorld(interpolationAlpha);
  requestAnimationFrame(render);
}
requestAnimationFrame(render);
```

The tick handler is deliberately synchronous: an authoritative tick must finish
before the next tick starts. To run simulation in a `Worker`, synchronously send
the tick command from the handler and consume shared snapshots or worker messages
from rendering. Do not make the handler `async` or return a Promise; the runtime
pauses the scheduler if that happens.

`maxCatchUpTicks` bounds work after a slow frame. Excess accumulated time is
discarded and reported as `ticksDropped`, while actual tick IDs remain contiguous.
This prevents a catch-up spiral without pretending skipped game logic ran.

```javascript
gameLoop.setTimeScale(5); // five times normal simulation time
gameLoop.pause();
gameLoop.step(1);         // executes on the next native frame
gameLoop.resume();

console.log(gameLoop.getState());
console.log(gameLoop.getStats());
```

## Semantic Agent Adapters

The built-in `mystral/agent` module exposes bounded, JSON-safe views of a live
game to the debug protocol and MCP server. The generic adapters work with any
engine; `exposeThree()` supplies scene, camera, and performance adapters for
Three.js-compatible objects without adding a Three.js dependency to the runtime.

```javascript
import { exposeThree } from "mystral/agent";

const agentIntegration = exposeThree({
  scene,
  camera,
  renderer,
  allowCameraMutations: true,
  allowSceneMutations: false,
  gameMetrics: () => ({ enemies: enemies.length, score })
});
```

For IDs that remain stable across runs, set `object.userData.agentId` on important
scene objects. Three.js UUIDs and generated IDs are accepted as fallbacks but are
reported with `stability: "session"`.

The default semantic entities are:

| Entity | Contents |
|--------|----------|
| `scene.main` | Paginated scene nodes, bounded by `depth` (0-8) and `limit` (1-500) |
| `scene.node` | One scene node selected by stable ID |
| `camera.active` | Active camera transform and projection settings |
| `performance.live` | Runtime heap/frame data, WebGPU counters, renderer data, and game metrics |

Scene mutation is only registered when `allowSceneMutations` is true. Camera
transform, look-at, and entity-focus actions are only registered when
`allowCameraMutations` is true. Keep both disabled for read-only agent sessions.

For a custom renderer, compose `exposeScene()`, `exposeCamera()`, and
`exposePerformance()` with synchronous snapshot/action functions. Each adapter
returns a `dispose()` function that removes its registrations.

## WebGPU

Full WebGPU API implementation via Dawn.

```javascript
// Request adapter
const adapter = await navigator.gpu.requestAdapter({
    powerPreference: "high-performance"
});

// Request device
const device = await adapter.requestDevice();

// Get canvas context
const context = canvas.getContext("webgpu");
const format = navigator.gpu.getPreferredCanvasFormat();

context.configure({
    device,
    format,
    alphaMode: "premultiplied"
});

// Create shaders, pipelines, buffers, etc.
const shaderModule = device.createShaderModule({
    code: `
        @vertex fn vs() -> @builtin(position) vec4f { ... }
        @fragment fn fs() -> @location(0) vec4f { ... }
    `
});
```

The Promise-based GPU operations support normal `async`/`await`. On Dawn,
pipeline compilation, queue completion, and buffer mapping are backed by native
asynchronous callbacks instead of blocking the JavaScript isolate:

```javascript
const pipeline = await device.createComputePipelineAsync({
    layout: "auto",
    compute: { module: shaderModule, entryPoint: "main" }
});

device.queue.submit([commandBuffer]);
await device.queue.onSubmittedWorkDone();
await readback.mapAsync(GPUMapMode.READ);
```

`createImageBitmap()` is also Promise-based and decodes images on the shared
native job system. See [Performance & Memory](/docs/guides/performance) for usage
and scheduling guidance.

## WebTransport

Low-latency client networking over HTTP/3 (QUIC), implemented natively via
[Cloudflare quiche](https://github.com/cloudflare/quiche). Supports unreliable
**datagrams** (UDP-like, for fast state updates) and reliable **streams**
(ordered, for chat, RPC, asset transfer) — the same `WebTransport` API you use in
the browser.

```javascript
const transport = new WebTransport("https://127.0.0.1:4433/echo");
await transport.ready;

// Unreliable datagrams
const writer = transport.datagrams.writable.getWriter();
await writer.write(new Uint8Array([1, 2, 3]));
const reader = transport.datagrams.readable.getReader();
const { value } = await reader.read(); // Uint8Array

// Reliable bidirectional stream
const stream = await transport.createBidirectionalStream();
const sw = stream.writable.getWriter();
await sw.write(new TextEncoder().encode("hello"));
await sw.close();
const sr = stream.readable.getReader();
const chunk = await sr.read(); // { value, done }

// Reliable unidirectional stream
const uni = await transport.createUnidirectionalStream();
await uni.getWriter().write(new Uint8Array([4, 5, 6]));

// Server-initiated streams
const incoming = transport.incomingUnidirectionalStreams.getReader();
const { value: serverStream } = await incoming.read();
```

Supported: `ready`/`closed`/`close()`, `datagrams`,
`createBidirectionalStream()`, `createUnidirectionalStream()`,
`incomingUnidirectionalStreams`, `incomingBidirectionalStreams`, and
`WebTransportError`.

> **Note:** WebTransport requires a build with quiche compiled in
> (`MYSTRAL_HAS_QUICHE`). The server URL must use `https://` with an explicit
> port. `serverCertificateHashes` (self-signed pinning) is not yet supported.
> See the [WebTransport guide](/mystralnative/docs/guides/webtransport) for full
> details, the test fixture, and architecture notes.

## Gamepad

Standard Gamepad API for controller input.

```javascript
// Connection events
window.addEventListener("gamepadconnected", (e) => {
    console.log("Connected:", e.gamepad.id);
});

window.addEventListener("gamepaddisconnected", (e) => {
    console.log("Disconnected:", e.gamepad.id);
});

// Poll gamepad state in game loop
function update() {
    const gamepads = navigator.getGamepads();
    for (const gamepad of gamepads) {
        if (!gamepad) continue;

        // Axes (sticks)
        const leftX = gamepad.axes[0];  // -1 to 1
        const leftY = gamepad.axes[1];
        const rightX = gamepad.axes[2];
        const rightY = gamepad.axes[3];

        // Buttons
        const a = gamepad.buttons[0].pressed;
        const b = gamepad.buttons[1].pressed;
        const x = gamepad.buttons[2].pressed;
        const y = gamepad.buttons[3].pressed;
    }
}
```

## Worker

Each Worker runs on its own OS thread with an independent JavaScript engine and
heap. Module workers can load JavaScript or TypeScript through the runtime module
resolver. JSON-compatible control data and typed binary views use `postMessage`;
shared game state is passed by handle without copying its backing memory.

```javascript
const worker = new Worker('./workers/simulation.ts', {
    type: 'module',
    name: 'simulation',
    maxMessages: 64,
    maxMessageBytes: 256 * 1024 * 1024,
    maxQueuedBytes: 384 * 1024 * 1024,
});

worker.onmessage = (e) => console.log("Result:", e.data);
worker.postMessage(21); // Result: 42
worker.terminate();
```

Blob workers and classic `importScripts()` remain supported. The worker loop is
event-driven and sleeps until a message or timer is due. Worker globals include
real `setTimeout`, `clearTimeout`, `setInterval`, `clearInterval`, and
`queueMicrotask`. Timers use a monotonic native clock and do not busy-poll.

`ArrayBuffer`, TypedArray, and `DataView` values preserve their binary type,
offset, length, and shared-backing relationships. A plain backing buffer can be
transferred with ownership semantics; the sender is detached when `postMessage`
accepts the transfer list:

```javascript
const vertices = new ArrayBuffer(4 * 1024 * 1024);
const positions = new Float32Array(vertices, 256, 1024);
worker.postMessage({ positions }, [vertices]);
console.log(vertices.byteLength); // 0
```

V8 moves a transferred backing store without copying and reconstructs
`positions` as a `Float32Array` in the receiver. Without a transfer list, the
runtime clones each distinct backing once and leaves the sender's buffer intact.
Transfer-list entries remain limited to plain `ArrayBuffer` objects. Duplicate
entries, non-`ArrayBuffer` entries, detached buffers, and raw shared-buffer views
are rejected. Use the shared APIs below for storage that remains attached in both
isolates.

Input and output user-message queues default to 1,024 messages, 16 MiB for one
message, and 64 MiB accumulated per direction. `maxMessages`,
`maxMessageBytes`, and `maxQueuedBytes` override those bounds per Worker;
`maxMessageBytes` cannot exceed `maxQueuedBytes`. Byte accounting includes wire
metadata and every attached backing, including a zero-copy transfer whose memory
is still retained by the queue. `postMessage()` throws `RangeError` when the
input queue is full or a message exceeds its individual limit. Worker-side
`postMessage()` throws when its output queue reaches the same configured bounds.
Use `SharedQueue` for sustained high-rate game traffic instead of relying on an
unbounded message backlog.

The runtime profiler exposes active Worker counts at sample start/end,
input/output queue bytes at sample end, lifetime queue high-water bytes, largest
message, and separate queue-full/message-too-large rejection counters for each
direction. A zero end value does not mean the queue was never pressured; inspect
the lifetime peak. Any nonzero rejection is backpressure that application code
must handle rather than a silently dropped message.

`terminate()` interrupts currently executing JavaScript on V8, so a
Worker stuck in a pure JavaScript loop cannot block normal runtime shutdown or
hot reload.

### Game-oriented WorkerPool

`mystral/worker-pool` keeps a fixed set of Workers alive. Its named task API
supports bounded queued rounds, static partitions, dynamically claimed chunks,
one-serialization native broadcast, and deterministic owner-side reductions.

`WorkerPool.create()` can be called in the main isolate or inside a first-level
Worker. The default runtime depth supports the intended simulation topology:
main/render at depth 0, an authoritative simulation Worker at depth 1, and the
pool's task Workers at depth 2. Pool Workers cannot create another generation of
Workers. The complete Worker tree shares one runtime worker limit and one
`SharedBufferRegistry`; terminating a parent interrupts and joins all descendants
before destroying the parent's JavaScript isolate.

Inside a Worker, nested `Worker` construction accepts module URL strings. Inline
`Blob` and classic-script sources remain available only from the main isolate.
`WorkerPool` already uses module URLs, so this does not restrict pool ownership.

```javascript
// simulation-pool.worker.mjs
import { exposeWorkerTasks } from 'mystral/worker-pool';
import { CharacterRows } from './CharacterRows.mjs';

exposeWorkerTasks({
    demography({ data, begin, end, isCancelled }) {
        const characters = CharacterRows.attach(data.characters);
        for (let row = begin; row < end; row++) {
            if ((row & 255) === 0 && isCancelled()) return;
            characters.birthDay[row] -= 1;
        }
        return end - begin;
    },
});
```

```javascript
// main.mjs
import { WorkerPool } from 'mystral/worker-pool';

const pool = await WorkerPool.create('./simulation-pool.worker.mjs', {
    size: 4,
    name: 'simulation',
});

const cancellation = new AbortController();
await pool.forEach(
    'demography',
    { characters: characters.handle() },
    {
        length: characters.length,
        grainSize: 512,
        signal: cancellation.signal,
    },
);
await pool.close();
```

`WorkerPool.create()` waits until every task module loaded, registered its
handlers, and acknowledged its pool index. Direct construction is intentionally
unsupported, so code cannot receive a partially initialized pool. Startup rejects
after 30 seconds by default rather than waiting forever when a task module fails
to register handlers; `startupTimeoutMs` changes that bound.

`parallelFor()`, `forEach()`, and `runNativeTask()` return Promises and can be
used directly with `await`. Task handlers may also be `async`; their Worker does
not publish a result or enter the round barrier until the returned Promise
settles.

`parallelFor()` defaults to dynamic scheduling. Workers claim chunks through a
shared atomic counter, which prevents one expensive contiguous partition from
holding the barrier. `schedule: 'static'` selects fixed partitions and stable
Worker-index result order. Built-in
reducers are `sum`, `count`, `min`, and `max`; a local reducer function is also
accepted and always consumes chunk results in chunk-index order.

Every task is registered by name with `exposeWorkerTasks()` and invoked by name.
Up to 64 waiting rounds are queued by default (`maxQueuedRounds` changes the
bound); a full queue rejects instead of growing without limit. A task error
rejects the active and queued rounds and closes the pool so partially completed
state is not reused.

Dynamic rounds containing only value data and shared handles use the native
broadcast path: the parent serializes the task once, then the registry publishes
the same immutable payload to every child queue. Per-chunk transferable
`ArrayBuffer` results are intentionally unsupported; write bulk output to shared
storage or return small reduction values.

`forEach()` is the mutation-only API. It avoids collecting per-chunk values:
every Worker decrements a shared atomic barrier and only the last Worker
publishes one completion message for the whole round. Use it when correctness is
represented by writes to shared state or single-writer command lanes. Its
options accept an `AbortSignal`. Aborting a queued round
removes and rejects it immediately. Aborting an active round sets a shared
atomic cancellation flag, stops additional chunk claims, waits for current
handlers and the pool barrier, then rejects. Handlers receive `isCancelled()`
for optional checks inside a large chunk. Cancellation is cooperative; use
`close()` when an executing isolate must be interrupted forcibly.

The package publishes TypeScript declarations from `types/mystral.d.ts`,
including task contexts, options, reducers, lifecycle methods, and result types.

### Shared game state

Schemas are ordinary modules imported by the main isolate and every Worker that
uses them:

```javascript
// CharacterRows.mjs
import { SharedTable } from 'mystral/shared';

export const CharacterRows = SharedTable.define('game.characters/v1', {
    templateIndex: 'u16',
    settlementId: 'u32',
    householdRef: 'u32',
    birthDay: 'i32',
});
```

```javascript
// main.mjs
import { CharacterRows } from './CharacterRows.mjs';

const characters = CharacterRows.create({ capacity: 200_000 });
const worker = new Worker('./simulation.worker.mjs', { type: 'module' });
worker.postMessage({ characters: characters.handle() });
```

```javascript
// simulation.worker.mjs
import { CharacterRows } from './CharacterRows.mjs';

self.onmessage = ({ data }) => {
    const characters = CharacterRows.attach(data.characters);
    characters.birthDay[0] = -10_000;
};
```

`SharedTable` uses fixed-capacity struct-of-arrays storage. `attach()` validates
the schema id, layout hash, capacity, and shared header. `definition.resize()`
performs an explicit migration into a new backing store; the old table stays
valid until it is released:

```javascript
// Stop writers and wait for the current round first.
const larger = CharacterRows.resize(characters, { capacity: 400_000 });
// Distribute larger.handle() and wait for Worker acknowledgements.
characters.release();
characters = larger;
```

`pool.resizeTable(CharacterRows, characters, { capacity })` applies the same
migration and rejects it while that pool has an active round. It does not manage
Workers outside the pool, so the application must still establish a global safe
point before retiring the old table. Attached TypedArray views are never replaced
transparently.

`SharedQueue.define()` creates a bounded single-producer/single-consumer ring
buffer. Payload fields can be `u8`, `i8`, `u16`, `i16`, `u32`, `i32`, `f32`, or
`f64`. Queue publication uses `Atomics`, while rows assigned exclusively to one
Worker can use normal TypedArray writes.

`SharedCommandBuffer.define()` creates one fixed-capacity lane per Worker. Each
Worker writes only to its own lane, avoiding multi-producer contention. After the
pool barrier, the simulation owner drains all lanes and may provide a stable key:

```javascript
import { SharedCommandBuffer } from 'mystral/shared';

const Commands = SharedCommandBuffer.define('game.commands/v1', {
    entity: 'u32',
    kind: 'u16',
    value: 'f32',
});
const commands = Commands.create({ workerCount: pool.size, laneCapacity: 4096 });

// Worker: commands.push(workerIndex, { entity, kind, value });
// Simulation owner, after the barrier:
const stableCommands = commands.drain({ sortBy: ['entity', 'kind'] });
```

Use a unique or sufficiently discriminating `sortBy` key when dynamic scheduling
must produce identical commit order across runs. `push()` returns `false` when a
lane is full. `clear()`, `size`, `handle()`, `attach()`, and `release()` follow
the same explicit lifetime model as the other shared primitives.

### Native tasks from Workers

Every Worker exposes the virtual `mystral/native-tasks` module. It submits a
registered C++ handler to the native priority `JobSystem` and resolves the
Promise back inside the calling isolate:

```javascript
import { runNativeTask } from 'mystral/native-tasks';

const result = await runNativeTask('game.path-batch', {
    requests: requestTable.handle(),
});
```

Native threads never access V8 handles. The bridge accepts value data and shared
handles, but rejects `ArrayBuffer` attachments. The native handler receives the
versioned wire payload and returns a wire payload or an error through
`NativeTaskResult`. Closing a Worker cancels its queued native generation and a
weak mailbox discards late completion safely. A Rust kernel can be called behind
a registered C++ adapter without changing Worker ownership.

The runtime-wide CPU execution budget is shared by Worker JavaScript and native
jobs. By default it reserves one logical CPU for main/render; embedders can set
`RuntimeConfig::maxCpuThreads`. This prevents persistent V8 Workers and the
native JobSystem from independently saturating all logical CPUs.

### Tradeoffs

- JavaScript objects and closures are private to their isolate.
- `postMessage` does not yet implement the browser structured-clone algorithm;
  payloads must be JSON-compatible, except for Mystral shared handles and
  explicitly transferred `ArrayBuffer` values.
- Rendering and WebGPU remain on the main thread.
- External shared backing stores and forced execution interruption are available
  on V8. Blocking `Atomics.wait()` is not exposed; use Worker messages or
  `SharedQueue` publication instead.

Run the fixed shared-memory benchmark with `mystral run benchmarks/worker-shared.mjs`
and the ownership-transfer benchmark with
`mystral run benchmarks/worker-transfer.mjs --no-sdl`. Compare fixed and dynamic
scheduling on an intentionally skewed workload with
`mystral run benchmarks/worker-scheduler-v2.mjs --no-sdl`.
